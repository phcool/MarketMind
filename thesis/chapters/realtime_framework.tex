\chapter{面向股票预测的多源信息融合框架设计}

\section{任务定义}

本文研究的核心任务是面向单只股票在给定分析日期下的短期走势方向预测。给定目标股票、分析日期以及该日期之前可见的历史行情和外部文本信息，系统需要判断该股票在未来 1 个、3 个和 7 个交易日后的价格方向，即相对分析日收盘价更可能上涨还是下跌。与精确涨跌幅回归相比，方向预测更适合本文当前的数据条件和实验目标，也更符合投研场景中先判断趋势、再进行风险控制和仓位管理的分析流程。

从输入信息看，本文框架主要包含两类信号。第一类是结构化行情信号，包括开盘价、最高价、最低价、收盘价、成交量、涨跌幅、换手率等 K 线特征；第二类是非结构化文本信号，包括投资者评论、新闻报道和机构研报等。行情信号能够反映已经形成的价格与量能状态，而文本信号能够提供市场情绪、事件冲击和专业分析逻辑。本文希望通过大语言模型将二者统一组织起来，形成更接近真实投研流程的多源信息融合预测框架。

\section{总体框架}

MarketMind 采用“多源获取、分支汇总、统一融合、方向预测”的整体框架。系统首先围绕目标股票和分析日期收集多源信息；随后对不同来源的信息分别进行清洗、筛选和摘要；最后将多源摘要与历史行情共同输入大语言模型，输出未来多周期方向判断和主要依据。

该框架可以分为四个层次。第一层是数据层，负责维护行情、评论、新闻和研报等原始数据。第二层是信息处理层，负责对高噪声、长文本或异构结构进行筛选和压缩。第三层是融合推理层，利用大语言模型对不同来源之间的一致性、冲突点和潜在因果关系进行综合判断。第四层是输出层，面向未来 1、3、7 个交易日生成方向预测和解释依据。

这种分层设计的主要原因在于，多源金融信息具有明显的长度差异和噪声差异。如果直接将所有原始评论、新闻和研报一次性输入模型，不仅会造成上下文浪费，也容易让模型被重复信息或低质量文本干扰。因此，本文采用分支摘要作为中间表示，使每类信息先被压缩为更稳定的结构化或半结构化内容，再进入最终预测环节。

结合当前仓库中的在线系统实现，上述总体框架可以进一步细化为一条可执行链路：首先根据股票代码生成实时信息快照，包括最近 50 条 comments、最近 10 条 news、最近 3 条 reports、最近 7 个交易日 K 线以及 SerpAPI 补充检索结果；随后由用户按需对 comments、news、reports 和 SerpAPI 结果分别触发 Summary；最后调用统一预测 Prompt，将 K 线与已生成的 news / reports / SerpAPI Summary 输入模型，输出未来 1 日、3 日、7 日方向判断。这样，论文中的框架不再只是离线方法描述，而是已经在在线原型系统中形成了“抓取—摘要—综合预测”的闭环。

\section{多源信息来源设计}

\subsection{行情信息}

行情信息是股票方向预测中最基础的数据来源。本文使用目标日期之前连续若干个交易日的 K 线作为结构化输入。对于监督微调主线，系统使用连续 7 个交易日的归一化 K 线预测第 8 个交易日方向；对于多源信息验证，系统使用目标日期及之前连续 5 个交易日的归一化 K 线，并预测未来 1、3、7 个交易日方向。

行情数据的作用主要有两个。第一，它提供价格和成交量层面的短期状态，例如趋势是否延续、是否出现放量反弹、是否存在连续下跌或高位震荡。第二，它为文本信息提供市场背景。相同的新闻或评论在不同技术形态下可能产生不同影响，因此模型需要同时看到行情状态和外部文本，才能形成更合理的短期判断。

\subsection{评论信息}

评论信息主要用于捕捉投资者情绪、讨论热度和短期市场叙事。论坛或股吧中的评论通常更新较快，能够较早反映散户投资者对热点题材、突发消息和价格波动的反应。但评论文本噪声较高，包含大量情绪宣泄、重复观点和非理性表达，因此不适合直接作为最终预测依据。

在框架设计中，评论分支的目标不是保留所有原始评论，而是提取其中的代表性观点和情绪共识。系统优先选择互动量较高或关注度较强的评论标题，再引导大语言模型归纳主要观点簇、情绪倾向和共识程度。这样可以将原始高噪声评论转化为更适合预测环节使用的“市场情绪摘要”。

\subsection{新闻信息}

新闻信息主要体现事件驱动特征。公司经营变化、行业政策、市场传闻、公告解读和突发风险通常会通过新闻报道进入市场信息流，并在短期内影响投资者预期。与评论相比，新闻文本结构更规范、事实性更强，但也可能存在报道角度差异、标题信息不足和发布时间滞后等问题。

在本文框架中，新闻分支主要用于提取与未来走势相关的事件线索。系统将目标日期之前可见的相关新闻组织为标题或摘要列表，要求模型归纳主要事件主题、媒体叙述倾向以及可能影响股价的关键信息。对于更完整的后续系统，新闻分支还可以采用两阶段方式：先基于标题筛选重要新闻，再读取正文进行二次摘要。

\subsection{研报信息}

研报信息主要提供专业分析和中长期逻辑。与评论和新闻相比，研报通常包含更完整的行业背景、公司基本面分析、业绩预期、风险提示和投资逻辑。虽然研报的时效性不一定最强，但其分析深度能够帮助模型补充更稳定的解释框架。

在框架设计中，研报分支用于提供高质量分析证据。系统选取分析日期之前较近的研报，对其核心观点、逻辑主线、潜在机会和风险因素进行摘要。最终预测时，研报摘要可以与新闻事件和评论情绪共同构成跨源证据，使模型不只依赖短期情绪，也能够参考相对专业的分析观点。

\subsection{Web API 检索信息}

除 comments、news 和 reports 外，本文进一步将外部 Web API 检索结果作为补充信息源接入框架。该部分的目标不是替代已有金融站点抓取，而是在实时查询阶段为目标股票补充更广泛的公开网页信息，例如最新媒体报道、公告解读、股价异动讨论、财报相关新闻以及面向个股的研究观点等。

在当前实现中，Web API 检索通过 SerpAPI 调用搜索引擎结果完成。系统围绕目标股票构造多组检索 query，包括“公司名 + 股票 + 最新消息”“公司名 + 股价 + 异动”“公司名 + 公告”“公司名 + 财报/业绩”“公司名 + 研报”以及“股票代码 + 公司名”等形式。每组 query 默认获取前 3 条结果，并对返回内容中的标题、链接、摘要、来源和日期进行抽取。为了减少不同检索意图之间的大量重复网页，系统在多组 query 返回后进一步以 URL 为主键进行统一去重与合并，最终形成一份补充搜索结果列表。

与新闻分支相比，Web API 检索信息具有两个特点。第一，其覆盖范围更广，不局限于单一资讯站点，因此更适合作为临时热点、媒体扩散信息和交叉验证证据的补充来源；第二，其噪声水平更高，不同 query 之间可能出现重复结果、弱相关结果或摘要质量不稳定的问题。因此，在框架中，Web API 检索结果不直接替代 news/report 分支，而是作为最后一层补充证据输入，在综合分析时帮助模型识别是否存在新的外部事件线索或证据支持。

\section{多源信息组织与 Prompt 设计}

为了使大语言模型稳定理解多源输入，本文采用统一的信息组织方式。每次预测输入都显式包含股票代码、分析日期、行情窗口、信息来源和对应摘要内容。不同信息源在 Prompt 中被分块展示，避免评论、新闻和研报混杂在一起造成语义边界不清。

评论分支 Prompt 强调观点聚类和情绪共识，要求模型识别主要讨论主题、情绪方向和共识强度。新闻分支 Prompt 强调事件与影响，要求模型总结当日或近期新闻中的核心事实和潜在利好利空因素。研报分支 Prompt 强调专业观点和风险提示，要求模型提取分析逻辑、投资机会和不确定性。Web API 检索分支则强调“补充线索与交叉验证”，重点保留搜索结果中的来源、日期和摘要信息，使模型能够快速判断补充网页是否为已有新闻和研报之外的新证据。综合预测 Prompt 则要求模型比较不同来源之间的一致与冲突，并结合 K 线状态输出未来多周期方向判断。

在当前实现中，不同分支的 Prompt 已经被明确区分为两类。第一类是分支 Summary Prompt，即分别针对 comments、news、reports 和 SerpAPI 网页正文生成集中观点总结。comments 的输入以标题、发布时间、点击量和评论量为主，强调从高互动讨论中归纳主要情绪与争议点；news 与 reports 的输入则保留标题、日期和正文内容，强调从事件与分析逻辑中提取共识观点；SerpAPI 部分在抓取正文之前会先经过规则过滤，剔除 PDF 链接，再对剩余普通网页执行正文抓取与总结。第二类是综合预测 Prompt，该 Prompt 不再直接消费所有原始文本，而是只接收最近 7 个交易日 K 线以及已生成的 news / reports / SerpAPI Summary。这样的设计有意将 comments 保留为辅助观察分支，而把更偏事实与分析性的三个摘要分支作为最终预测的主要外部信息来源。

为了与本文训练数据集中的 CoT 风格保持一致，综合预测 Prompt 还沿用了“先推理、后输出结论”的格式约束。模型需要先在固定的“思维链开始/结束”标记之间分析 K 线趋势、量能变化以及外部信息摘要，再分别输出未来 1 个交易日、3 个交易日和 7 个交易日的最终判断。当前在线原型系统中，最终判断使用“上涨/下跌”表述；而离线训练与强化学习数据中使用单字“涨/跌”标签。二者在语义上完全对应，本质上都属于方向二分类输出，不允许输出“震荡”“持平”等中间类别。这样一方面保证了在线预测与离线监督数据在输出风格上的一致性，另一方面也便于后续将在线预测结果进一步解析、记录和评估。

这种 Prompt 设计的关键在于“先压缩、再融合”。分支摘要阶段降低了原始文本长度和噪声，综合预测阶段则把不同来源的信息统一到同一个决策问题中。相比简单拼接原始文本，该方法更便于控制上下文长度，也更容易让模型给出可解释的预测依据。

\section{预测输出设计}

本文将预测输出设计为面向未来 1 日、3 日和 7 日的多时间尺度结果。当前在线原型系统中，每个时间尺度的最终结论只保留“上涨”或“下跌”两个类别，以便与离线数据集中的涨跌标签保持一致；同时，模型需要在输出最终结论之前先给出简短分析过程，用于说明判断依据主要来自哪些 K 线特征、新闻事件、研报逻辑或外部搜索证据。

在离线实验中，模型输出可以进一步解析为“涨”或“跌”标签，用于计算准确率和分周期表现。在在线应用中，解释依据具有更重要的辅助价值，因为用户不仅关心模型预测结果，也需要判断该结果是否来自可靠证据、是否存在信息源冲突，以及是否需要结合人工判断进一步验证。当前网站中的 Predict 按钮正是按照这一思路实现：先由用户按需生成各分支 Summary，再在统一预测入口中输出多周期方向结论，从而把原始信息浏览、分支总结和综合判断组织为一套完整交互流程。

\section{本章小结}

本章从方法层面提出了 MarketMind 多源信息融合预测框架。该框架将股票短期方向预测建模为行情状态、市场情绪、新闻事件和专业分析共同作用下的多源决策问题。通过分支摘要和统一融合，系统能够将高噪声、异构化的金融文本转化为可供大语言模型推理的结构化输入，并输出未来 1、3、7 个交易日的方向预测和解释依据。下一章将进一步说明这些数据如何被构造成训练与评测样本，以及如何用于基础模型的监督微调。
