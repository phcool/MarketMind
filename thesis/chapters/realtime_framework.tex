\chapter{面向股票预测的多源实时信息增强框架设计}

\section{任务定义与总体框架}

本文面向股票短期走势预测任务，设计一个由多源信息获取、文本内容缓存、分支信息汇总和大语言模型综合预测组成的实时信息增强框架。给定目标股票和分析日期，系统需要在该时点下获取与该股票相关的评论、新闻、研报以及后续可扩展的 Web 检索信息，并结合最近若干交易日的行情状态，输出未来 1 日、3 日和 7 日的涨跌方向判断。

从输入来源看，本文框架主要包含四类信息：第一类是投资者评论，即来自股吧或论坛等平台的用户讨论；第二类是新闻信息，即与目标股票相关的新闻标题和正文；第三类是机构研报，即卖方或研究机构发布的研究报告；第四类是 Web API 检索结果，用于补充实时外部信息。当前工程实现已经完成 Comments、News、Reports 三类来源的数据抓取、缓存和大模型摘要，Web API 检索模块作为扩展接口预留。

从系统结构看，框架采用“分支汇总—统一融合”的设计。Comments 分支强调市场情绪和群体观点，News 分支强调事件和媒体叙事，Reports 分支强调专业逻辑和风险分析。每一类信息先独立进行筛选与摘要，得到结构化或半结构化的中间表示；随后系统将各分支摘要与行情特征共同输入大语言模型，由模型完成跨源证据整合、冲突识别和未来走势预测。该设计避免直接把大量原始文本一次性输入模型造成上下文冗余，同时也便于不同信息源独立迭代。

\section{系统整体流程设计}

\subsection{预测触发机制}

系统以“股票 + 日期”为基本预测触发单元。在在线演示实现中，用户首先在前端页面选择目标股票和分析日期；后端根据股票代码和日期，从本地 CSV 数据文件和正文缓存目录中查询对应信息。股票池由 \path|config/stocks.json| 统一维护，并通过 \path|scripts/stock_universe.py| 加载为行业、股票名称、股票代码和市场类型的结构化列表。当前界面主要面向 A 股六位数字代码进行展示和预测。

在 \path|scripts/stock_daily_dashboard.py| 中，系统通过 \texttt{query\_data} 函数完成预测时点的信息定位：Comments 和 News 选取分析日期当日的数据，Reports 则选取分析日期之前或当天最近的若干篇研报。该机制体现了真实预测场景中的时间约束，即模型在某一时点进行判断时，不应读取未来信息。

\subsection{实时信息获取流程}

信息获取流程由离线批处理采集和在线按需读取两部分组成。离线批处理阶段主要负责将外部网页数据写入本地统一存储；在线预测阶段则优先读取本地缓存，并在必要时补充抓取正文内容。

在批处理采集阶段，\path|scripts/fetch/fetch_market_data.py| 提供统一入口，可通过 \texttt{--mode comments}、\texttt{--mode news}、\texttt{--mode report} 或 \texttt{--mode all} 分别抓取评论、新闻和研报列表。评论数据来自东方财富股吧接口，系统按照股票分页抓取帖子列表，并保留帖子标题、发布时间、点击数和评论数等字段。新闻数据来自东方财富新闻搜索结果，系统根据股票名称检索相关新闻，并从 URL 中解析发布时间。研报数据来自新浪财经研报列表，系统按照股票代码抓取研报标题、链接和发布日期。

为了提高采集稳定性，系统在抓取过程中引入了 checkpoint 机制。Comments 抓取按公司记录已完成页码，News 和 Reports 抓取按平台记录已完成股票集合；当采集中断后，下一次运行可以从 checkpoint 恢复，减少重复请求和人工干预。同时，采集脚本对请求失败、空页面、重复页面和过旧研报设置了停止条件，避免无限分页或无效抓取。

在内容增强阶段，\path|scripts/fetch/fetch_content_to_disk.py| 负责将新闻正文和研报正文落盘到 \path|Content/news/| 与 \path|Content/report/| 目录。新闻正文按照月份和 URL 哈希组织文件路径，研报正文按照 URL 哈希组织文件路径。缓存文件中保留 URL、标题、日期等元信息，并以分隔符之后的正文作为后续摘要输入。对于 HTTP 404 或解析为空的页面，脚本会删除对应 CSV 行或跳过无效内容，从而保持数据源的可用性。

\subsection{多源信息汇总流程}

多源信息汇总采用分支式设计。对于 Comments，系统先按点击量和评论数对当日帖子进行排序，再截取高曝光帖子输入模型，要求模型输出观点簇、情绪强度、共识程度以及正面、中性、负面概率。对于 News，系统将当日新闻标题列表输入模型，要求模型总结主要主题、媒体叙述倾向，并列出与未来走势关联较强的新闻编号。对于 Reports，系统优先读取缓存正文；若缓存不存在，则尝试即时抓取研报正文，再将最近若干篇研报按时间倒序拼接输入模型，要求模型总结主要观点、逻辑主线、风险因素和机会因素。

在工程实现中，三个分支分别对应 \texttt{/summarize\_stream}、\texttt{/summarize\_news\_stream} 和 \texttt{/summarize\_reports\_stream} 三个流式接口。系统通过 DashScope 的 OpenAI-compatible API 调用 Qwen 模型，并采用 SSE 方式把模型生成内容实时返回前端。这种流式设计可以降低用户等待感，也便于在页面中分别观察不同信息源的摘要结果。

\subsection{模型推理与结果输出流程}

在完成信息汇总后，系统将摘要文本和最近 7 个交易日 K 线数据共同输入预测模型。当前在线预测实现中，\texttt{/predict} 接口接收前端传入的摘要文本，并调用 \texttt{\_load\_recent\_kline} 从 \path|exports/quotes.csv| 中读取截至分析日期的最近 7 个交易日行情，包括开盘价、最高价、最低价、收盘价、涨跌幅和成交量等字段。随后，\texttt{\_predict\_with\_llm} 将舆情总结与 K 线数据拼接成统一 Prompt，要求模型分别输出未来 1 日、3 日和 7 日的方向、置信度及主要依据。

从框架角度看，预测输出不是单一标签，而是“方向 + 置信度 + 证据说明”的组合。方向用于对应最终涨跌判断，置信度用于表达模型对该判断的自我评估，证据说明则用于解释模型主要依据哪些评论、新闻、研报或行情变化形成判断。这种设计有利于后续人工审查，也便于将模型输出、输入快照和未来真实走势共同保存为训练样本。

\section{多源信息获取模块设计}

\subsection{Comments 获取模块}

Comments 获取模块负责采集投资者论坛中的股票讨论信息。当前实现以东方财富股吧为主要来源，通过股票代码分页请求股吧帖子列表接口。每条评论记录保留 URL、帖子 ID、股票代码、帖子标题、发布时间、点击数和评论数，最终写入 \path|exports/comments.csv|。

考虑到股吧评论数量较大且噪声较多，系统在存储和查询阶段均进行截断控制。在写入阶段，\path|scripts/fetch/csv_io.py| 中的 \texttt{merge\_append\_comments} 会按 URL 去重，并在全表层面对每个“股票—自然日”最多保留 200 条评论，排序依据主要是点击数。在在线查询阶段，\texttt{query\_data} 会进一步筛选目标股票和目标日期的数据，并最多返回 500 条候选记录；摘要 Prompt 中再按照点击量和评论数排序，实际提供给模型的上限为 200 条。

该模块的设计重点不是保留所有评论，而是优先保留市场可见度较高的讨论。点击量和评论数虽然不能直接等价于预测价值，但能够近似反映讨论的传播范围和关注度。通过这种排序截断策略，系统可以在控制上下文长度的同时，让模型优先看到更可能代表当日市场关注点的评论内容。

\subsection{News 获取模块}

News 获取模块负责采集目标股票相关新闻。当前实现中，系统使用股票名称作为关键词，在东方财富新闻搜索页面中抓取新闻列表，保存 URL、股票代码、标题和日期到 \path|exports/news.csv|。由于新闻搜索页面需要浏览器环境加载，脚本使用 Playwright 启动无头 Chromium，并按页采集新闻条目。为避免无效分页，脚本设置最大页数和连续空页停止规则。

新闻列表只是第一阶段信息。为了获得更完整的事件上下文，系统进一步通过 \path|scripts/fetch/fetch_content_to_disk.py| 抓取新闻正文，将正文缓存到 \path|Content/news/{YYYY-MM}/{url_hash}.txt|。正文解析时主要提取网页中的正文容器和段落内容，并将 URL、标题、日期等元数据与正文一起写入缓存文件。后续构造历史样本或在线摘要时，可以通过 URL 哈希定位正文，避免重复访问外部网站。

在当前在线摘要实现中，News 分支主要使用当日新闻标题进行主题总结，并要求模型列出与未来走势关联较强的新闻编号。该设计对应架构中的第一阶段 News 总结。后续可在此基础上加入 Top-K 新闻正文增强：先由标题级摘要筛选重要新闻，再读取对应正文缓存或按需抓取正文，最后生成更充分的 News 分支总结。

\subsection{Reports 获取模块}

Reports 获取模块负责采集和利用机构研报信息。当前实现从新浪财经研报列表中按股票代码抓取研报标题、URL 和发布日期，并写入 \path|exports/report.csv|。由于研报正文网页存在编码和结构差异，正文抓取单独封装在 \path|scripts/fetch/fetch_report_content.py| 和 \path|scripts/fetch/fetch_content_to_disk.py| 中，主要解析 \texttt{div.blk\_container} 或 \texttt{div.content} 下的段落文本。

在线预测时，Reports 分支遵循“只使用预测时点之前信息”的约束。系统从 \path|exports/report.csv| 中筛选目标股票在分析日期之前或当天发布的研报，并按日期倒序选取最近若干篇。当前 dashboard 展示与摘要默认使用最近 3 条研报；架构规划中也可以根据实验设置调整为最近 2 篇。对于每篇研报，系统优先读取 \path|Content/report/| 中的缓存正文；若不存在缓存，则调用正文抓取函数即时获取并写入缓存。

Reports 摘要 Prompt 会将多篇研报按新到旧排列，并要求模型总结主要观点、逻辑主线、风险因素、潜在机会和投资者结论。与 Comments 的情绪共识不同，Reports 分支更强调专业分析逻辑和风险收益权衡，因此其输出更适合作为综合预测阶段的高质量证据来源。

\subsection{Web API 检索模块}

本节预留 Web API 检索模块。该模块计划用于在预测时补充评论、新闻和研报之外的实时外部信息，例如搜索引擎结果、公告补充、行业政策信息或市场热点解释。当前论文与工程实现中暂不展开该模块的具体接口、字段结构和调用策略，后续可在统一输入模板中将其作为第四类信息源接入。

\section{多源信息统一组织与汇总设计}

\subsection{输入模板设计}

为了让大语言模型稳定理解多源信息，本文采用统一的输入模板组织方式。每次模型调用均显式给出股票名称、分析日期、信息来源、原始条数或选取条数，并按照固定字段排列具体内容。例如 Comments 输入中包含帖子序号、点击数、评论数和标题；News 输入中包含 \texttt{news\_id} 与新闻标题；Reports 输入中包含报告序号、标题、日期、URL 和正文片段。

统一模板的作用主要体现在三个方面。首先，它降低了模型理解输入结构的难度，使模型能够区分不同文本的来源和权重。其次，它便于对长文本进行截断和压缩，例如研报正文超过上限时只保留前若干字符，并明确标注截断信息。最后，它为后续离线样本构建提供了可复现格式，使同一股票、同一日期的输入快照能够被保存、回放和用于监督微调。

在历史数据构建方面，\path|scripts/dataset/build_daily_stock_text_dataset.py| 已经体现了类似思想。该脚本按日期和股票遍历样本，读取指定时间范围内的 News 和 Reports 正文缓存，并输出包含 \texttt{stock}、\texttt{time}、\texttt{news} 和 \texttt{reports} 字段的 CSV 文件。其中 News 默认保留最近 10 条，Reports 默认保留最近 3 条，为后续离线实验提供了统一的数据组织基础。

\subsection{汇总提示词设计}

汇总提示词采用分源定制策略，而不是对所有信息源使用同一个通用摘要指令。Comments 分支要求模型输出 JSON，并包含观点簇、情绪强度、共识程度和全局情绪概率。这种结构化输出有助于把高噪声评论转化为可比较、可拼接的情绪共识表示。News 分支要求模型总结主题、媒体叙述倾向、重要新闻编号和投资者结论，重点提取事件驱动信息。Reports 分支则要求模型概括主要观点、共同风险、潜在机会和结论要点，重点保留专业分析逻辑。

在综合预测阶段，提示词进一步要求模型把摘要信息和最近 7 个交易日 K 线数据共同考虑，并分别输出未来 1 日、3 日和 7 日的走势预测。通过将不同预测周期显式列为输出项，模型可以针对短期事件冲击和稍长周期的趋势延续分别作出判断，避免把不同时间尺度混为一个结论。

\section{股票短期走势预测输出设计}

本文将预测输出设计为面向未来 1 日、3 日和 7 日的多时间尺度结果。每个时间尺度至少包含三个要素：方向、置信度和主要依据。方向用于表明模型判断未来价格更可能上涨还是下跌；置信度使用 0 到 1 的数值表示模型对判断的相对确定性；主要依据则以一句或数句自然语言说明模型参考了哪些信息。

这种输出形式兼顾了机器评估和人工理解。对于离线实验，可以将方向结果映射为涨跌二分类标签，与未来真实涨跌进行比较；对于在线使用，置信度和依据可以帮助使用者判断模型是否只是给出机械标签，还是确实综合了评论情绪、新闻事件、研报逻辑和近期行情。后续如果引入强化学习或偏好优化，也可以围绕方向准确性、理由一致性、风险提示完整性和跨源证据利用程度构造奖励或偏好信号。

\section{本章小结}

本章围绕面向股票短期预测的多源实时信息增强框架进行了设计说明。该框架以目标股票和分析日期为触发条件，围绕 Comments、News、Reports 和预留 Web API 四类来源构建信息获取链路，并通过本地 CSV 与正文缓存实现可复现的数据管理。在信息处理层面，系统采用分支摘要方式，将高噪声评论、事件型新闻和专业研报分别转化为适合大语言模型理解的中间表示；在预测层面，系统将多源摘要与最近 7 个交易日行情共同输入模型，输出未来 1 日、3 日和 7 日的方向、置信度和依据。该框架为后续模型训练、离线实验和在线部署奠定了工程基础。
